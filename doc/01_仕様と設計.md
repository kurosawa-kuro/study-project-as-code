# リーダースキル資格 仕様と設計

> 資格戦略の仕様 + 教材設計。プロジェクト全体像を伝える土台。
> 要件レベルの変更時に更新する。矛盾時は [`02_移行ロードマップ.md`](./02_移行ロードマップ.md) を正とする。

## 共通前提

- **目的**: リモート技術リードとしての信用補強に必要な資格戦略を、**判断基準つきの参考書**として整理する。単なる資格紹介集ではなく、「今の実務状態なら何を取り、何を取らず、何を学習だけで回収するか」を決めるための判断辞書を作る。
- **入力 / 出力**:
  - 入力 = 現在の実務状況、技術主軸、現場評価、各資格の守備範囲、学習コスト、職務経歴書での見せ方。
  - 出力 = リーダースキル資格の戦略仕様書（Markdown）。資格ごとの役割定義、優先順位、採否基準、学習順、見せ方、破綻条件を含む。
- **担当範囲**: リーダースキル資格の方針整理と参考書設計。対象は PMLE / PSPO I / CAPM / PSM I / Databricks 基礎資格の扱い。具体的な講座購入、受験申込、職務経歴書本文の最終版作成は対象外。
- **ユースケース**:
  1. 「次に何の資格を進めるべきか」を、技術主軸を崩さずに判断する。
  2. 「PSPO I は受験すべきか」「CAPM は重すぎないか」のような迷いを、条件ベースで切り分ける。
  3. 職務経歴書や面談で、資格の意味を PM 専業寄りに見せず、技術リード強化として説明する。
- **開発環境**: Markdown 執筆とレビュー。`doc/` 配下のドキュメントを同一変更で更新する。
- **本番環境**: この文書自体が参照元となる。将来的な配布形態はリポジトリ閲覧を前提とする。
- **言語 / フレームワーク**: 日本語 Markdown。アプリフレームワークは持たない。
- **設定管理**: 非機密は `env/config.yaml`、ローカル秘密情報は `env/secret.yaml`、共有・本番秘密情報は Doppler（`doppler.yaml`）
- **CI/CD**: 現時点では未整備。将来的に Markdown lint / リンク整合チェックを追加する余地がある。

## 1. アーキテクチャ

このプロジェクトの設計対象は「アプリケーション」ではなく「資格判断の構造」である。情報の流れは「実務状況と候補資格の整理 → 判断単位への分解 → 最終方針の固定」とする。

```text
実務状況（現場評価・技術主軸・不足観点）
        │
        ▼
[1] 戦略の世界観（PM 専業化しない / 技術主軸維持）     ← §1.1
        │
        ▼
[2] 資格を判断単位に分解（取る / 学習のみ / 保留）       ← §1.2
        │
        ├─ 資格比較表（役割と採否基準）                   ← §2.2
        ├─ 判断シグナル（この状態なら何を疑うか）          ← §2.3
        └─ 学習フロー（どの順で語彙と証明を積むか）        ← §2.4
        │
        ▼
[3] 運用設計（職務経歴書・説明方法・破綻回避）           ← §2.5
```

### コンセプト

- **資格の参考書ではなく判断辞書である**: 各資格の概要説明だけで終わらせず、「どの条件で取るか」「どの条件なら学習だけでよいか」「どの条件なら今は保留か」まで定義する。
- **主語は資格名ではなく意思決定である**: 章の中心は「PSPO I を受験するか」ではなく、「価値判断の語彙をどう回収するか」「PM 管理観点をどう外部証明するか」といった判断になる。
- **技術主軸を絶対に崩さない**: この戦略は PM 専業化の準備ではなく、技術リードの再現性を高める補強策である。
- **実務評価を土台にする**: 資格でゼロから信用を作るのではなく、すでにある高評価を外部証明と説明言語で補強する。

### 1.1 戦略の世界観

各判断は次の世界観に従う。

| 世界観 | 意味 |
|---|---|
| 技術主軸優先 | 主戦場は PMLE、GCP、ML、MLOps、Python、BigQuery、Kubernetes、Databricks |
| PM 専業化しない | 資格取得の目的は管理職転向ではなく、技術リード信用補強 |
| 実務評価を先に置く | すでに現場で出ている評価を、資格で説明しやすくする |
| 軽い資格を集めすぎない | 肩書きが増えても単価源泉が弱くなるなら逆効果 |
| 語彙と証明を分けて考える | PSPO I は語彙獲得、CAPM は外部証明というように役割を分離する |
| 学習コストは技術学習に従属 | PMLE や技術学習を圧迫する資格戦略は採用しない |

### 1.2 対象とする判断単位

この文書で対象とする最低限の判断単位は以下。

- リーダースキル資格の目的を「信用補強」に固定してよいか
- PSPO I を受験するのではなく学習のみで回収してよいか
- CAPM を PM 管理観点の本命として採用してよいか
- PSM I を今は保留してよいか
- Databricks 基礎資格を技術横幅の回収先として後段に置いてよいか
- 職務経歴書でどう見せれば PM 専業志望に見えないか

## 2. 機能仕様

この教材を構成する主要機能は 5 つある。ここでいう機能はソフトウェア機能ではなく、判断を成立させるためのコンテンツ部品である。

### 2.1 資格判断テンプレート

各資格は以下の 8 項目で統一して整理する。

1. **資格の一言定義** — 何を証明する資格か
2. **この資格で補える不足** — 実務のどの弱点を補強できるか
3. **今の状態で選ぶ条件** — 現時点で取得価値が高い条件
4. **今の状態で選ばない条件** — 学習だけで十分な条件、あるいは後回しにすべき条件
5. **ひっかけパターン** — 「取った方がよさそう」に見えるが実は優先度が低い状況
6. **代替手段** — 受験ではなく学習、実務実績、別資格で回収できるか
7. **ロードマップ上の位置** — 先にやるべきか、後でよいか、保留か
8. **最後に覚える一文** — その資格の役割を一言で固定する

例として、PSPO I は次の型で整理する。

| 項目 | 内容 |
|---|---|
| 資格の一言定義 | Product Owner 観点の価値判断・優先順位・Backlog 整理の基礎語彙を扱う資格 |
| 補える不足 | 顧客価値、優先順位、ステークホルダー調整の言語化 |
| 選ぶ条件 | 実務でこの観点が弱く、外部証明が必要な場合 |
| 選ばない条件 | 実務評価で代替でき、CAPM 前の語彙獲得で足りる場合 |
| ひっかけ | 「軽いから取りやすいので先に受験する」という発想 |
| 代替手段 | Scrum Guide、模擬問題、Product Owner Open で学習回収 |
| ロードマップ位置 | PMLE の後、CAPM の前に学習のみ |
| 覚える一文 | PSPO I は証明より語彙回収に使う |

### 2.2 資格比較表

似た価値に見える資格でも、補う不足は異なる。比較表で切り分ける。

| 比較軸 | 最低限必要な比較 |
|---|---|
| 価値判断の語彙 vs 管理観点の証明 | PSPO I vs CAPM |
| Scrum 運営の基礎証明 vs 今の優先課題 | PSM I vs CAPM |
| 技術本命 vs 補助資格 | PMLE vs リーダー系資格全般 |
| 技術横幅の維持 vs 管理寄りの見え方 | Databricks 基礎資格 vs リーダー資格追加 |
| 学習のみで十分か、受験すべきか | PSPO I、PSM I、CAPM |

現在の基準は以下。

| 資格 | 位置づけ | 判断 |
|---|---|---|
| PMLE | 技術本命 | 最優先で取得 |
| PSPO I | 価値判断・優先順位・Backlog の学習材料 | 学習のみ |
| CAPM | 予算・人月・スコープ・リスク管理の外部証明 | 取得する |
| PSM I | Scrum チーム運営の基礎証明 | 保留 |
| Databricks 基礎資格 | データ / ML 基盤の横幅補強 | CAPM 後に検討 |

### 2.3 判断シグナル辞典

現状の状態や悩みから、何を疑うべきかを引く辞典を置く。

| 状況 / 悩み | 疑うべき判断 |
|---|---|
| 顧客価値や優先順位の語彙が弱い | PSPO I を学習で回収する |
| 予算・人月・スコープ・リスク管理の説明が弱い | CAPM を本命にする |
| チーム運営の基礎証明を問われた | PSM I を再検討する |
| 技術主軸がぼやけそう | リーダー資格の追加取得を止める |
| PM 専業に見られたくない | PMLE と Databricks を主軸に残す |
| 実務評価を次案件で説明しづらい | CAPM と職務経歴書の言語化を強化する |
| 学習時間が不足している | PSPO I は受験せず学習のみで回収する |

### 2.4 学習フローチャート

この戦略の基本フローは以下。

```text
技術主軸をまず強めたい？
 ├─ Yes → PMLE を最優先
 │   └─ その後、価値判断の語彙が足りない？
 │       ├─ Yes → PSPO I を学習のみで回収
 │       └─ No  → CAPM の準備へ
 │
 └─ No → 例外ケース。原則としてこの流れは採用しない

CAPM で補いたい不足がある？
 ├─ Yes → 23 時間教育を満たして取得
 └─ No  → 受験を急がず、実務要件を再確認

その後も技術横幅を強めたい？
 ├─ Yes → Databricks 基礎資格を検討
 └─ No  → 現状維持
```

PSPO I と CAPM の接続は次の通り整理する。

```text
PSPO I で「作る価値と順番」を学ぶ。
CAPM で「人月・予算・リスクで実行管理する」。
```

### 2.5 運用テンプレート

資格戦略は、取得だけでなく見せ方まで含めて成立する。最低限、以下をテンプレート化する。

- **成功条件**:

| 成功条件 | 内容 |
|---|---|
| 技術主軸を維持する | PM 専業に見せない |
| 現場評価を言語化する | 派遣先・エンド双方からの高評価を職務経歴書に反映する |
| CAPM を外部証明として使う | 予算・人月・スコープ・リスク管理を補強する |
| PSPO I を学習で回収する | 資格取得にこだわらず、語彙獲得に使う |
| Databricks で技術横幅を作る | ML / データ基盤側へ軸足を戻す |

- **破綻条件**:

| 破綻条件 | 内容 | 対策 |
|---|---|---|
| リーダー資格を集めすぎる | 技術主軸がぼやける | PSPO I 受験、PSM I 取得は保留する |
| CAPM を重くしすぎる | PMLE や技術学習を圧迫する | 合格点突破型で進める |
| PM 専業に見せる | 技術リードとしての単価源泉が弱くなる | 「CAPM を持つ技術リード」として見せる |
| PSPO I で満足する | 本命の PM 管理観点が補強されない | PSPO I は学習のみ、CAPM を本命にする |
| 現場評価を書かない | 資格だけの人に見える | 実務評価を職務経歴書に明記する |

- **職務経歴書テンプレート**:

悪い見せ方:

```text
CAPM取得予定。
PSPO I学習中。
PM業務に対応可能。
```

良い見せ方:

```text
リモート環境において、GCP / Vertex AI / BigQuery / Pythonバッチ / 検索改善に関する技術判断・改善推進を担当。
派遣先およびエンドクライアント双方から、報告・論点整理・推進姿勢について高評価を獲得。
今後はCAPMにより、工数・スコープ・リスク・ステークホルダー管理のPM基礎を体系化し、リモート技術リードとしての再現性を高める。
```

## 3. 設計上の決定事項

### 3.1 採用したアプローチと理由

- **判断辞書フォーマット（§2.1）を採用** — 単なる資格紹介では「何を取るべきか」が決まらない。現状に応じた採否判断まで書く必要がある。
- **資格名中心ではなく意思決定中心で構成する** — 「PSPO I の説明」より「価値判断の語彙をどう回収するか」の方が、実際の迷いに直結する。
- **技術主軸を固定した上で資格を従属させる** — PMLE、GCP、ML、データ基盤を主戦場に固定することで、リーダー資格が単価源泉を壊さないようにする。
- **PSPO I を学習、CAPM を本命とする二段構えを採用** — 価値判断の語彙と PM 管理観点の外部証明を分離した方が、過剰受験を避けつつ必要な補強ができる。
- **職務経歴書での見せ方まで設計対象に含める** — 資格戦略は取得だけで完結せず、次案件でどう説明されるかまで含めて初めて機能する。

### 3.2 採用しなかったアプローチ

以下は明示的に避ける。

1. **リーダー資格を数で積む** — 複数資格で見栄えはよくなっても、技術主軸がぼやけるなら逆効果。
2. **PSPO I を本命として受験する** — 今の状態では実務評価で代替可能な部分が多く、費用対効果が弱い。
3. **PSM I を先に取る** — Scrum 運営の基礎証明としては有効だが、現時点の不足には CAPM ほど刺さらない。
4. **CAPM を重い長期プロジェクト化する** — PMLE や実務学習を圧迫するほど重くすると、本来の主戦場を損なう。
5. **資格だけで信用を作ろうとする** — 実務評価を書かずに資格だけ並べると、かえって「実績の薄い人」に見えやすい。

## 4. 非機能要件

本プロジェクトは静的な参考書ドキュメントであるため、ソフトウェア的な性能要件よりもコンテンツ品質要件を重視する。

- **性能 / スケール**: 該当なし。静的 Markdown 文書として読めることを前提とする。
- **セキュリティ / 認証**: 機密情報を本文に書かない。`env/secret.yaml` はコミットしない。
- **可用性 / 監視**: 該当なし。リポジトリ閲覧可能であることを前提とする。
- **判断一貫性**: 各資格を §2.1 のテンプレートで整理し、採否理由の書き方を統一する。
- **説明可能性**: 「なぜ取るか」「なぜ取らないか」を第三者が読んで追える状態にする。
- **鮮度**: 実務状況、目標案件、資格制度の変化があれば見直す。特に CAPM の受験要件や学習時間見積もりが変わった場合は即更新する。
- **検証可能性**: 実務評価、資格の守備範囲、受験要件などは根拠が追える情報に基づいて記述し、推測で断定しない。
- **実用性**: 読んだあとに「次に何をやるか」が決まる構成を保つ。単なる感想文や一般論にしない。

## 5. 現時点の結論

現時点の最適方針は以下。

```text
PMLE
↓
PSPO I 学習のみ
↓
CAPM 取得
↓
Databricks 基礎資格
```

補足は以下。

- PMLE は技術本命
- PSPO I は受験必須ではなく、価値判断の語彙回収に使う
- CAPM はリモート技術リードとしての PM 管理観点を外部証明する本命
- PSM I は今回は保留
- Databricks 基礎資格は技術横幅維持のために後段で検討する

CAPM の前提情報は次の通り。

| 項目 | 内容 |
|---|---|
| 実技 | なし |
| 問題数 | 150 問 |
| 試験時間 | 180 分 |
| 形式 | 選択式中心 |
| 受験方法 | Pearson VUE 会場またはオンライン監督 |
| 日本語対応 | あり |
| 受験要件 | 高卒相当以上 + 23 時間以上の PM 教育 |

23 時間教育要件の成立条件:

```text
CAPM 対応の 23 時間以上の PM 講座を受講し、
修了証を保存する。
```

講座選定時は以下を確認する。

- CAPM Exam Prep
- 23 contact hours
- CAPM eligibility requirement
- Certificate of Completion
- 修了証発行
