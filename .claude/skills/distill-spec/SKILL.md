# 仕様蒸留

ChatGPT 等との生ブレストや貼り付けメモを、確定した **仕様 (01_requirements.md)** と **基礎設計 (02_architecture.md)** に整理分離する。生ログは消さず `docs/archive/` へ退避し、出典・候補・思考過程は素材として残す。

## いつ使うか

- `docs/01_requirements.md` などに生のブレスト・会話ログ・未整理メモが貼り付けられていて、テンプレ（目的/範囲/ユースケース等）が埋まっていないとき。
- 1 つの文書に「何を満たすか(WHAT)」「どう作るか(HOW)」「ただの候補・素材」が混在しているとき。

## 線引き

| 行き先 | 入れるもの |
|---|---|
| `01_requirements.md`（仕様 / WHAT） | 目的・対象/非対象・ユーザー・ユースケース(UC 表)・制約 |
| `02_architecture.md`（基礎設計 / HOW） | 概要・成長フェーズ・ディレクトリ/構成要素・境界 |
| `docs/archive/<topic>-brainstorm.md`（素材） | 生ログ全文・出典・候補一覧・思考過程（権威は最下位） |
| `docs/tasks/backlog/`（未確定） | 仕様化に足る根拠がまだ無い論点・調査対象 |

## 手順

1. **境界を見つける**: 生ログの範囲と、既存の確定記述／テンプレ節（`# 01 要件`〜「関連タスク」等）の開始行を特定する。
2. **生ログを退避**: `docs/archive/<topic>-brainstorm.md` を作り、生ログ全文をそのまま移す。冒頭に「退避元・蒸留先・権威は archive 最下位」の注記を付ける。**削除はしない。**
3. **仕様を蒸留 (01)**: 生ログから「何を満たすか」だけを抜き、目的・範囲(対象/非対象)・ユーザー・ユースケース(UC-001…の表)・制約に落とす。
4. **基礎設計を蒸留 (02)**: 「どう作るか」を抜き、概要・成長フェーズ・構成要素・境界に落とす。既存の構成要素行（AGENTS.md / CLAUDE.md / docs/tasks 等）は残す。
5. **素材は残す**: 仕様でも設計でもない候補一覧・出典は archive に残し、着手対象なら `docs/tasks/backlog/` に task 化する。
6. **drift を作らない**: `docs/00_index.md` の権威順位と各文書の「関連タスク」節を壊さない。01 と 02 と archive で同じ定義を重複させない。

## ルール

- 生ログを削除しない。必ず archive へ退避してからテンプレを書き直す。
- 01 = WHAT(要件)、02 = HOW(基礎設計) を混ぜない。実装手段を 01 に、目的を 02 に書かない。
- 推測で要件・設計を作らない。生ログに根拠があるものだけ蒸留し、曖昧な点は backlog へ回す。
- 正本を複製しない（同じ定義を複数文書に置かない）。
- 秘密情報・API キー・実在の個人/案件データ・アカウント情報を書かない。
