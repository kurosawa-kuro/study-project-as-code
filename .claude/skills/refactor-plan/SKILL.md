# リファクタ計画

現状の挙動を保ったまま、散らばった類似ロジックのリファクタを計画する。

## レビュー観点

- 重複・類似の処理を棚卸しする。
- 共通化すべきものと、分けたままにすべきものを判断する。
- 単一責務に違反するモジュールを分割する。
- ドメインロジックを、外部 I/O・DB・フレームワーク・UI・運用の関心事から分離する。
- 該当する箇所では port / adapter / repository の境界を明確にする。
- 構造変更の前に、現状の挙動を保つテストを追加する。
- もう割に合わない抽象化を減らす。

## ルール

- 似ているという理由だけでコードを共通化しない。
- 曖昧な共有ユーティリティの寄せ集めを作らない。
- デザインパターンそのものを目的にしない。
- 現状の挙動を確認する前に構造を変えない。
- テスト戦略なしに大きな変更を提案しない。

## 出力

- 重複の棚卸し
- 共通化 / 分離のままにする判断
- 境界の再設計案
- テスト計画
- 段階的なリファクタ計画
- 削除または隔離の候補
