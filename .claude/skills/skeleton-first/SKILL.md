# スケルトンファースト

中規模以上の作業では、AI に「期待動作」「置き場所」「実装」を同時に考えさせない。
仕様テスト、ビジネスロジックレス・スケルトン、テスト接続修正、実装を分け、各フェーズで AI の探索範囲を狭める。

これは工程を増やすための手順ではない。初手の token は少し増えるが、責務ズレ、構造やり直し、テスト改ざん、再読込の往復を減らし、結果的に品質と token 制御を安定させるための戦略である。

## 基本サイクル

標準サイクル:

```text
仕様書
↓
仕様ベースのテストコード作成
↓
ビジネスロジックレス・スケルトン作成
↓
テストコードの接続修正
↓
RED 確認
↓
実装
↓
GREEN 確認
↓
テスト不備修正
↓
再 GREEN 確認
```

役割分担:

- 仕様書 = 何を実現し、何を変えないか
- 仕様テスト = 何を満たすか
- スケルトン = どこに置くか
- テスト接続修正 = テストを現実の構造に接続する
- RED 確認 = 未実装なので正しく失敗する状態を作る
- 実装 = TODO をどう満たすか
- GREEN 確認 = 仕様テストを通す

## いつ使うか

使う:

- Clean Architecture / DDD 寄りの責務分離
- Ports / Adapters、gateway、worker、batch、DB Adapter
- 既存コードへの追加実装
- リファクタリング、Adapter 整理、共通化
- 後方互換を保つ改修
- テスト不足の既存コード改善

スキップする:

- 小さな修正
- 1〜2ファイルで完結する単純 CRUD
- 文言、設定値、ドキュメントだけの変更

小さい修正では `仕様 -> テスト -> 実装` の方が速い。スケルトン工程が過剰なら使わない。

## フロー判断

| 対象 | 推奨フロー |
|---|---|
| 小さい修正 / 単純 CRUD | 仕様 -> テスト -> 実装 |
| 中規模以上 / 既存構造への追加 | 仕様 -> テスト -> スケルトン -> テスト接続修正 -> RED -> 実装 -> GREEN |
| リファクタ / 責務分離 | 仕様 -> テスト -> スケルトン -> RED -> 実装 -> GREEN |
| Adapter 整理 / 共通化 | 仕様 -> テスト -> Port/Adapter スケルトン -> 接続修正 -> 実装 |
| 後方互換・期待動作を先に固定 | 仕様 -> テスト -> スケルトン -> テスト接続修正 -> 実装 |

## 各フェーズ

### 1. 仕様書

挙動だけでなく、変更範囲と禁止事項を明示する。

- 目的
- 入力、出力
- 成功条件、失敗条件
- 非対象
- 後方互換
- 変更禁止領域
- 従うべき既存設計

仕様書を巨大にしすぎない。毎回の token を抑えるため、今回の変更に必要な範囲へ絞る。

### 2. 仕様ベースのテストコード

先に期待動作を固定する。まだスケルトンがないため、最初のテストは理想仕様ベースでよい。

許容すること:

- この時点では compile 不可でもよい
- import path、型名、DI 構造が仮でもよい
- mock / fake の差し込み位置が仮でもよい

避けること:

- 実装詳細に依存した assertion
- 内部構造を過剰に縛るテスト
- 仕様未確定の挙動を固定するテスト
- 外部 API や実 DB に直接依存するテスト

### 3. スケルトン（ビジネスロジックレス）

実処理を実装せず、責務配置、ファイル構成、interface、依存方向だけを固定する。

許可:

- ファイル作成
- 型定義
- interface / trait 定義
- 関数シグネチャ
- DTO、Error 型
- UseCase の空実装
- Repository Port
- Outbound Adapter の空実装
- DI 配線
- TODO コメント
- compile を通す最小 stub

禁止:

- ビジネス判定
- DB 更新
- HTTP/API 呼び出し
- AI API 呼び出し
- 既存挙動の変更
- 勝手な責務追加
- 既存 Adapter の置き換え
- 不要な抽象化

スケルトンは薄くてよい。ロジックを入れ始めると、レビュー対象が増えて逆効果になる。

### 4. テストコードの接続修正

スケルトン後、テストが想定していた関数名、UseCase 名、DI 構造、Port / Adapter 構成、mock の差し込み位置がズレることがある。
このズレは仕様のズレではなく接続面のズレである。ここでテストを「実行可能だが、未実装なので失敗する」状態へ持っていく。

修正してよいもの:

- import path
- module path
- 型名、関数名との接続
- mock / fake の接続
- fixture
- DI 配線
- テスト実行環境
- compile error
- setup / teardown

修正してはいけないもの:

- 仕様上の期待値
- assertion の強さ
- 異常系ケース
- 後方互換テスト
- 失敗すべきケースを成功扱いにすること
- 本来検出すべき回帰を見逃すこと

期待値を変える必要が出た場合は、テスト修正ではなく仕様変更として扱う。

```text
仕様書更新
↓
テスト期待値更新
↓
スケルトン / 実装修正
```

### 5. RED 確認

理想状態:

- compile は通る
- テストは実行できる
- ビジネスロジック未実装なので失敗する

RED が確認できると、実装フェーズでは「既存スケルトン構造を変更せず、TODO だけ埋めて GREEN にする」と依頼できる。

### 6. 実装

スケルトンの TODO だけを埋める。

制約:

- 新しいファイルを勝手に追加しない
- 新しい責務を勝手に追加しない
- 新しい層を勝手に追加しない
- 新しい外部依存を勝手に追加しない
- 既存 Adapter を勝手に置き換えない
- テストの期待値を勝手に変更しない
- 後方互換を維持する
- 差分を最小化する

許可:

- TODO 実装
- テストを通すための最小実装
- 異常系の処理
- 必要最小限のエラーハンドリング
- mock / fixture / setup の不備修正
- compile error 修正

### 7. GREEN 確認とテスト不備修正

GREEN 後も、テスト側の不備が見つかることはある。修正してよいのは接続、fixture、mock、setup / teardown などの不備であり、実装都合で期待値を弱めてはいけない。

最後の「テストコード動作修正」はテスト改ざんになりやすい。ここを混同するとサイクル全体が破綻する。

## 注意点

### テスト接続修正は仕様変更ではない

テストコード動作修正は、テストを現実の構造に接続する工程である。期待動作を実装都合へ合わせる工程ではない。

### RED を省略しない

RED なしで実装へ進むと、テストが本当に仕様を検出しているか分からない。未実装なので失敗する状態を一度作る。

### スケルトンを厚くしない

スケルトンに判定条件、変換処理、DB 更新、API 呼び出しが入ったらやりすぎ。スケルトンは AI の探索空間を狭めるための構造制約であり、実装の下書きではない。

### 仕様書を長くしすぎない

仕様書が巨大だと初手 token が増えすぎる。目的、変更対象、非対象、成功条件、失敗条件、後方互換、禁止事項へ絞る。

### 期待値変更は仕様書から始める

期待値を変えたい場合は、先に仕様書を更新する。テストだけを変更して GREEN にすると、AI が「通るテスト」を作るだけになる。

## token 制御の考え方

この方式は token を先払いして、後続の迷走 token を削る。

token が減る箇所:

- 実装後の構造ミス修正が減る
- 設計レビューと実装レビューを分離できる
- AI が新しい層、責務移動、Adapter 置換をしにくくなる
- テスト修正の許可範囲が明確になる
- 実装依頼を「TODO だけ埋める」に圧縮できる

token が増えるケース:

- 小さすぎる変更に使う
- スケルトンを厚く作りすぎる
- 仕様書を毎回巨大化する
- テスト接続修正が期待値改ざんになる

## 依頼テンプレート（仕様テスト工程）

```text
仕様書に基づいて、先に仕様ベースのテストコードを書いてください。

この時点ではスケルトンが未確定なので、import path や DI 配線が仮でもかまいません。
ただし、期待値は仕様から導き、実装都合に依存させないでください。

含めるもの:
- 正常系
- 異常系
- 境界条件
- 既存仕様維持
- 回帰テスト

避けるもの:
- 実装詳細に依存した assertion
- 外部 API / 実 DB への直接依存
- 仕様未確定の挙動の固定
```

## 依頼テンプレート（スケルトン工程）

```text
まだ実装しないでください。
仕様書と仕様テストに基づいて、ビジネスロジックレスのスケルトンだけを作成してください。

目的:
- 実装前に責務配置と変更範囲を固定する
- 既存設計に沿ったファイル構成を確定する
- 後続のテスト接続修正と実装を安全に進める

許可:
- ファイル作成
- 型定義
- interface / trait 定義
- 関数シグネチャ
- DTO / Error 型
- DI 配線
- TODO コメント
- compile を通す最小 stub

禁止:
- ビジネス判定
- DB 更新
- HTTP/API/AI 呼び出し
- 既存挙動の変更
- 勝手な責務追加
- 既存 Adapter の置き換え
- 不要な抽象化

出力:
- 追加・変更するファイル一覧
- 各ファイルの責務
- 各型・関数の役割
- 実装フェーズで埋める TODO 一覧
- 既存設計との整合性
```

## 依頼テンプレート（テスト接続修正 / RED 確認）

```text
スケルトンに合わせて、テストコードを実行可能な状態へ接続してください。

修正してよいもの:
- import path
- module path
- 型名 / 関数名との接続
- mock / fake の接続
- fixture
- DI 配線
- compile error
- setup / teardown

変更してはいけないもの:
- 仕様上の期待値
- assertion の強さ
- 異常系ケース
- 後方互換テスト

目標:
- compile が通る
- テストが実行できる
- 未実装 TODO により RED になる
```

## 依頼テンプレート（実装工程）

```text
前フェーズで作成したスケルトン構造を変更せず、TODO 部分だけを実装してください。

禁止:
- 新しいファイル追加
- 責務移動
- 新しい層の追加
- 新しい外部依存の追加
- 既存 Adapter の置き換え
- テスト期待値の変更
- assertion の弱体化

許可:
- TODO 実装
- テストを通すための最小実装
- 必要最小限のエラーハンドリング
- mock / fixture / setup の不備修正
- compile error 修正

最後に出力:
- 実装内容の要約
- 変更ファイル一覧
- テスト結果
- 仕様との差分
- 未解決事項
```
