# PM / Leader Skill Model

Project as Code を、PM / リーダー業務へ適用するためのスキルモデル。

元素材:

- `/Users/kurosawa/Dev/pm-leader-skill/01-upper-lower-pm-model.md`

## 位置づけ

`project-as-code` は、プロジェクト情報を Markdown / YAML / Issue / コードとして構造化し、AI と人間が共に参照・更新できる状態を維持するフレームワーク。

`pm-leader-skill` は、その中でも PM / リーダーとしての市場価値、役割分解、説明文脈を扱う素材。

この文書では、PM / リーダースキルを PaC の正規モデルとして扱う。

## 上半分PM / 下半分PM

| 区分 | 主な責務 | PaC での主な成果物 |
|---|---|---|
| 上半分PM | 目的管理、課題管理、戦略管理、戦術管理 | context、requirements、decisions、specs |
| 下半分PM | 品質管理、進捗管理、メンバー管理、タスク管理 | tests、tasks、issues、runbooks |

## 上半分PM

未定義な状態から、何をやるべきかを決める力。

主な行動:

- 目的を定義する。
- 課題を構造化する。
- 制約と前提を明文化する。
- 戦略と選択肢を比較する。
- 何をやらないかを決める。
- PoC、設計、実装可能な skeleton へ落とす。

PaC での対応:

| 行動 | 対応する PaC |
|---|---|
| 顧客課題を聞く | Context as Code |
| 要件へ分解する | Requirement as Code |
| 選択肢と却下理由を残す | Decision as Code |
| 受入条件を定義する | Test as Code |
| 実装前の構造を固定する | Skeleton as Code |

## 下半分PM

決まったものを壊さず進める力。

主な行動:

- 進捗を管理する。
- タスクを分解する。
- 品質を確認する。
- メンバーや関係者と調整する。
- 未解決事項を追跡する。
- 完了条件を満たしたか確認する。

PaC での対応:

| 行動 | 対応する PaC |
|---|---|
| タスク化する | Issue as Code |
| 完了条件を固定する | Test as Code |
| 作業証跡を残す | docs/tasks |
| 判断のブレを防ぐ | Decision as Code |
| 再利用可能な手順にする | templates / runbooks |

## 黒澤さんのポジショニング

黒澤さんの強みは、下半分PMだけではなく、上半分PM寄り。

表現としては、次が強い。

> 未定義な AI / ML / データ基盤課題を、課題定義、技術構造、PoC、設計、実装、運用設計まで接続できる PM / アーキテクト。

避ける表現:

- 進捗管理できます。
- タスク管理できます。
- 技術もできます。

より強い表現:

- 課題定義から技術実装まで一本で通せる。
- AI / データ基盤の PoC を、設計・評価・運用まで接続できる。
- 上半分PM + アーキテクト + 実装者として動ける。

## PaC との接続

Project as Code は、上半分PMの暗黙知を成果物へ変換するための型。

```text
曖昧な相談
  -> context
  -> requirements
  -> decisions
  -> tests
  -> skeleton
  -> issues
  -> implementation
  -> verification
```

この流れにより、PM / リーダーの判断がチャットや口頭に散らばらず、AI と人間が再利用できる形で残る。

## CAPM / PMP の扱い

CAPM / PMP は主役ではなく、上半分PMの実務を世間の共通言語へ翻訳する補助線。

| 領域 | 黒澤さん実務との相性 |
|---|---|
| 統合管理 | 高い。PoC企画、設計、実装計画に対応する |
| スコープ管理 | 高い。AI / ML 課題のスコープ化に対応する |
| 品質管理 | 高い。評価基盤、テスト、精度管理に対応する |
| リスク管理 | 中程度。技術リスク、スケジュールリスクに対応する |
| 調達管理 | 低い。主役ではない |

資格は、実務価値を説明するための言語化補助として扱う。

## 破綻条件

- 下半分PMだけに見られる。
- 「技術もできるPM」で止める。
- 資格を主役にする。
- AIに判断を丸投げし、目的・課題・制約を文書化しない。
- タスクだけ増やして、Decision as Code を残さない。
