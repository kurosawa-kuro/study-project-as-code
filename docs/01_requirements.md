# 01 要件

## 目的

プロジェクト情報（前提・要件・判断・成果物・タスク）を Markdown / YAML / Issue / コードとして構造化し、**AIと人間が共に参照・更新できる状態を維持する**。

解決する課題は3つ。

| # | 課題 | 現状の問題 |
|---|------|-----------|
| 1 | プロジェクト情報の分散 | 口頭・議事録・チャットに散在し、AIに渡せる文脈がない |
| 2 | 意思決定の属人化 | 判断理由が残らず、同じ議論を繰り返す |
| 3 | 上流のボトルネック | 開発速度が上がるほど「何を・どの粒度で作るか」が詰まる |

### このリポジトリ自身の位置づけ

owner は新米リーダー・PM・コンサル見習いであり、その成長に関連するプロジェクトはリポジトリを分散させず、本リポジトリ（project-as-code）を親として `src/` 配下の子プロジェクトで統合管理して育てる（[ADR-002](adr/ADR-002-monorepo-growth-policy.md)）。

| 子プロジェクト | 役割 |
|---|---|
| `src/leader-skill/` | リーダースキル資格戦略と学習教材（CAPM / PSPO I） |
| `src/pmbok/` | PMBOK を軸とした PM 知識体系の正本整理 |
| `src/project-as-code/` | PaC フレームワーク自体を持ち出し可能な成果物として育てる |

---

## 範囲

**対象**

- コンサル・PM・リーダーが関与する中〜大規模プロジェクト
- AIエージェントに作業を委譲する開発プロセス
- 本番対応・リリース管理フェーズへの移行を伴うプロジェクト

**非対象**

- 単発の小規模タスク（1日完結）
- コード実装そのもの（PaCは実装の前段を扱う）
- PMBOKの全成果物を生成する目的での利用（重すぎて運用停止する）

---

## ユーザー

| ロール | PaCでの役割 | 主な操作 |
|--------|------------|---------|
| コンサル / PM | 設計・導入判断 | Context・Decision・Scopeを定義する |
| テックリード / リーダー | 運用・Issue分解 | Skeleton・Issue・Testを設計する |
| AIエージェント | 参照・生成 | `project_context.md` を読んで成果物を生成する |
| プロジェクトメンバー | 更新・実行 | Issue・Backlogを更新し実装を進める |
| PMO | 管理・報告 | リスク・課題・承認フローを参照する |

## ユースケース

| ID | ユースケース | 成功条件 |
|----|------------|---------|
| UC-001 | プロジェクト立ち上げ時にPaC構造を初期化する | 最小テンプレート（8ファイル）がgitに入り、AIが読める状態になる |
| UC-002 | 会議後に曖昧さをAIで検出しドキュメントへ反映する | 曖昧ポイントが分類され、ヒアリング質問に変換される |
| UC-003 | AIが `ai_context.md` を読んで仕様・Issue・テスト観点を生成する | 生成物が人間レビューに耐える粒度になっている |
| UC-004 | 意思決定をADRとして記録する | 判断理由・却下理由・未決事項が残り、後から参照できる |
| UC-005 | Issueを1時間〜半日粒度に分解してAI実装へ渡す | AIが1Issueを完結させ、人間レビューで合格できる |
| UC-006 | 本番対応前にリスク・影響範囲・承認フローを整理する | PMO・PMがそのまま管理資料に転記できる粒度で出力される |
| UC-007 | 技術課題をPM/PMO向けに翻訳して渡す | 現状・課題・対応案・影響範囲・判断が必要な点が揃っている |

## 非機能要件

| 観点 | 要件 |
|------|------|
| 鮮度 | 決定事項は会議後24時間以内にドキュメントへ反映する |
| 一元管理 | 同じ定義を複数ファイルに重複させない |
| AIアクセス可能性 | `ai_context.md` は常にAIが読める状態にある |
| 軽量性 | 運用できる最小ファイル数を維持する（PMBOKの全成果物は作らない） |
| トレーサビリティ | 要件 → ADR → Issue → テストが追跡できる |

## 参照フレームワーク

ユースケースに直接マッピングできるフレームワークのみを記載する。全体像は別途参照。

| フレームワーク | 対応UC | 用途 |
|---|---|---|
| イシューツリー | UC-002, UC-003 | 曖昧さと論点を構造化して分解する |
| So What? / Why So? | UC-004, UC-007 | 事実から示唆を出す／根拠を掘る |
| ピラミッドストラクチャー | UC-003, UC-007 | 結論→理由→根拠で出力を構造化する |
| WBS | UC-005 | 作業を階層分解し、Issue粒度に落とす |
| MoSCoW | UC-005 | Issueの優先度を Must / Should / Could / Won't で分類する |
| RACI | UC-006 | 役割と責任の境界を明示する |
| RAIDログ | UC-006 | Risks / Assumptions / Issues / Dependencies を管理する |
| As-Is / To-Be | UC-003, UC-007 | 現状と理想状態の差分を整理する |
