# ADR-002: リポジトリを分散させず project-as-code を親として育てる

- Status: Accepted
- Date: 2026-07-06

## Context

owner は新米リーダー・PM・コンサル見習いとしてキャリアをシフトアップ中であり、その成長に関連する知識・戦略・フレームワークが複数のリポジトリ（旧 `study-leader-skill` 等）に分散し始めていた。分散すると、ハーネス・運用ルール・判断記録が二重管理になり、学習途上の身では横断参照も統合的な育成もできなくなる。

## Decision

- リポジトリを分散させない。本リポジトリ（`study-project-as-code`）を親とし、リーダー・PM・コンサル成長に関連するプロジェクトはすべて `src/` 配下の子プロジェクトとして育てる。
- 現在の子プロジェクト:

| 子プロジェクト | 役割 |
|---|---|
| `src/leader-skill/` | リーダースキル資格戦略と学習教材（CAPM / PSPO I）。ADR-001 で吸収 |
| `src/pmbok/` | PMBOK を軸とした PM 知識体系の正本整理 |
| `src/project-as-code/` | PaC フレームワーク自体を持ち出し可能な成果物として育てる |

- 新しい関連テーマが生まれた場合も、独立リポジトリを新設せず `src/` 配下に子プロジェクトとして追加する（吸収手順は ADR-001 を踏襲）。
- 親は PaC の運用モデル（ハーネス・タスク運用・判断記録）を提供し、子はコンテンツに集中する。

## Alternatives Rejected

- **テーマごとの独立リポジトリ**: 学習途上では管理コストが成長を上回る。ハーネス・ルールの二重管理と横断参照の断絶を招くため却下。
- **親を汎用モノレポ名に改名**: PaC フレームワーク自体がこのリポジトリの中核成果物であり、「PaC で自分の成長プロジェクトを運用する」構図そのものに価値があるため、project-as-code を親の顔として維持する。

## Consequences

- 子プロジェクトの追加・吸収は ADR で記録する。
- 親のハーネス・運用ルールの改善は全子プロジェクトに波及する（統合管理の利点）。
- リポジトリが大きくなった場合の分割は、実際に運用が破綻し始めてから再検討する（早すぎる分割はしない）。
