# ADR-003: オリジナリティは意図して出さず、デファクトのフレームワークを忠実に組み合わせる

- Status: Accepted
- Date: 2026-07-06

## Context

owner は新米リーダー・PM・コンサル見習いであり（ADR-002）、本リポジトリとその子プロジェクトは PM・リーダー・コンサルスキルの成長基盤である。学習途上で独自フレームワークや独自用語を作ると、(1) デファクトとの差分が説明できず外部（面談・実務・資料）で通用しない、(2) 独自概念の保守コストが発生する、(3) AI との協働でも共通言語の恩恵（事前学習済み知識）を失う。

## Decision

**オリジナリティは意図して出さない。PMBOK / WBS / PDCA / MECE / ピラミッドストラクチャー等、デファクトのフレームワークを忠実に「組み合わせる」のが黒澤流。**

- 本リポジトリと全子プロジェクト（`src/leader-skill/` / `src/pmbok/` / `src/project-as-code/`）の文書・成果物・運用は、デファクトのフレームワークとその標準用語で記述する。
- 独自性は「どのフレームワークを・どの優先順位で・どう組み合わせるか」の選択にのみ置く。フレームワーク自体の改変・独自命名はしない。
- 採用するフレームワークの優先順位と用途の正本は `docs/01_requirements.md#参照フレームワーク` に置く。
- 新しいフレームワークの採用・優先順位の変更は `docs/01_requirements.md` を更新し、大きな方針変更は ADR に残す。

## Alternatives Rejected

- **独自フレームワーク・独自用語の考案**: 差別化に見えるが、学習途上では検証不能な我流の固定化になる。外部との共通言語も失うため却下。
- **フレームワークの部分改変（○○流アレンジ）**: 忠実さが失われると「デファクトを使える」という証明価値が消える。組み合わせの選択のみに独自性を残す。

## Consequences

- 文書レビュー時の観点に「デファクト用語からの逸脱がないか」が加わる。
- `src/pmbok/` は PMBOK を、`src/project-as-code/` は各フレームワークの組み合わせ方を、それぞれ標準に忠実な形で正本化する。
- AI エージェントへの指示も標準用語で行い、独自概念の定義文書を増やさない。
