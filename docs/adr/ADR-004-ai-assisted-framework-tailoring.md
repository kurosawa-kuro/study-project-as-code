# ADR-004: フレームワークの組み合わせ調整（テーラリング）は生成AIに委譲し、人間は判断基準を握る

- Status: Accepted
- Date: 2026-07-07
- 関連: [ADR-003](ADR-003-defacto-framework-policy.md)（デファクト忠実主義）の運用モデルを定める

## Context

ADR-003 で「デファクトのフレームワークを忠実に組み合わせる」ことを決めたが、フレームワークは単体では強くても、複数を雑に混ぜると衝突する（RACI と DACI の責任者二重化、PMBOK・Scrum・OKR・RAID の全部盛りによる過剰管理、MECE・イシューツリー・ピラミッドの抽象度衝突など）。この組み合わせ調整（テーラリング）を人間の知能だけで案件ごとに安定運用するには限界がある。

## Decision

黒澤流の完全定義を次のとおりとする。

> 黒澤流とは、デファクトのフレームワークを忠実に採用し、互いにカニバリズムを起こさないように組み合わせ、意図してフレームワーク層でオリジナリティを出さない実務設計思想である。独自性は、対象に応じた組み合わせ・重み・削減と、それを Project as Code として再利用可能な形に落とし込む点にある。

運用上の役割分担:

- **人間（owner）が握るもの**: 目的・制約・判断基準・破綻条件
- **生成AIに委譲するもの**: デファクトの選定・重み調整・削減・再構成・既存 PaC 資産の再利用
- **成果物**: Project as Code として残し、次回以降の判断資産にする
- **テーラリングの原則**: 固定セットではなく、対象（案件・役割）ごとにフレームワークの濃淡を変える。カニバリズム（役割重複・抽象度衝突）を起こす組み合わせは片方を主と決める

## Alternatives Rejected

- **生成AIに自由発明させる**: デファクトから外れて独自理論化する。AI は「発明者」ではなく「選定・調整・再利用の支援者」に限定する（ADR-003 と同根）。
- **全部盛りの固定セット**: PaC が重くなり運用が死ぬ。対象別の濃淡調整を前提とする。
- **人間だけでテーラリング**: 案件横断で一貫性を保てず、毎回ゼロから組み直しになる。

## Consequences

- テーラリングの具体構造（採用フレームワーク一覧と重み、対象別テーラリング方針、カニバリズム回避ルール、役割別プロファイル）は `src/project-as-code/` の実装候補とする。実装判断は `docs/tasks/backlog/implement-framework-tailoring-skeleton.md`。
- 人間が判断基準を渡さずに AI へ丸投げした場合（AI が一般論で組み合わせる）、および PaC 資産を更新せず実態と乖離した場合（台帳化）は、この方針の破綻条件である。
- カニバリズム回避の初期判断例: RACI vs DACI は主を一つ選ぶ / OKR vs KPI は目標と計測で分離 / MECE は原則・イシューツリーは成果物 / PMBOK は管理言語・Scrum はデリバリー手法 / ADR はアーキ判断・decision-log は業務/PM 判断。
