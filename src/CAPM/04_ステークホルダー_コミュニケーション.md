# ステークホルダー・コミュニケーション

## 1. この章で切れる選択肢

- 全員に同じ報告を流す
- 相手ごとに必要な判断材料を整理する
- 進捗だけ読み上げる
- 論点、選択肢、依頼事項まで含めて伝える

## 2. 技術の最短定義

ステークホルダー管理は、「誰が何を気にしているか」を整理し、関係者ごとの期待値をそろえることである。
コミュニケーション管理は、「何を、誰に、いつ、どの深さで伝えるか」を設計することである。

リモート環境では、技術力そのものより、この 2 つの品質が信頼感に直結しやすい。
作業している事実より、相手が判断しやすい状態を渡せるかが重要になる。

## 3. 今の状態で選ぶ条件

- 顧客、派遣先、開発メンバーで必要情報が違う
- 高評価を受けているが、再現可能な説明力に変えたい
- 報告はしているが、論点整理の型が弱い
- リモートで「見えていない不安」を減らしたい

## 4. 今の状態で選ばない条件

- 情報量が多ければ十分と思う
- 進捗報告を作業日記にする
- 判断してほしいことを曖昧にしたまま会議へ出る

## 5. ひっかけパターン

- 技術的に正しいので説明は最小限でよいと思う
- リスクが確定するまで待ってから共有する
- すべての相手に同じ資料で済ませる
- 課題を出すが、相手に何を判断してほしいかが抜ける

## 6. 代表パターン

### 6.1 相手ごとに必要な情報

| 相手 | 主に気にすること | 必要な伝え方 |
|---|---|---|
| 顧客 | 影響、優先順位、意思決定 | 選択肢と影響を簡潔に |
| 派遣先 / 上位 | 安定推進、リスク、報告品質 | 論点と見通しを整理して |
| 開発メンバー | 実行条件、依存関係、役割 | 具体的で行動可能な形で |

### 6.2 良い報告の型

良い報告は、最低限次の 4 点で構成される。

1. 今どこまで進んでいるか
2. 何が論点か
3. 放置した場合どうなるか
4. 何を判断してほしいか

### 6.3 悪い報告と良い報告

悪い報告:

```text
いま調査中です。少し遅れるかもしれません。
```

良い報告:

```text
API 仕様確認待ちのため、実装着手が 1 日後ろ倒しになる見込みです。
このままだと金曜レビューに影響するため、
1. 先に代替実装を進める
2. レビュー日を後ろへずらす
のどちらかを今日決めたいです。
```

## 7. 実務へのつなぎ方

### 7.1 期待値調整の考え方

- 相手が知りたい粒度は同じではない
- 顧客には判断材料、メンバーには実行条件、上位には見通しが必要
- 「全部説明する」より「相手が決められる形で渡す」が優先

### 7.2 リモート環境での加点行動

- 問題が小さいうちに論点化する
- 次アクションを自分で仮置きしてから相談する
- 会議前に、何を決める場なのかをはっきりさせる

### 7.3 CAPM 学習としての回収ポイント

この章で覚えたいのは、会議術そのものではない。
「報告」「期待値調整」「意思決定支援」を、管理観点の言葉で整理できることが重要である。

## 8. 最後に覚える一文

コミュニケーション管理は、情報を渡すことではなく、相手が次の判断をできる形に整えて渡すことである。
