# 実務ケースで覚える CAPM

この章は、各概念を単語で覚えるのではなく、実務でどう使うかに戻すためのケース集である。

## ケース 1: 追加要望が来た

### 状況

検索改善案件で、顧客から「この画面のフィルタも今 Sprint に入れたい」と要望が来た。
メンバーは 2 名、すでにレビュー待ちタスクが 2 本ある。

### ありがちな弱い反応

- とりあえず引き受ける
- 現場で頑張って吸収する
- 影響説明を後回しにする

### CAPM 的に切る観点

- スコープ: 今回の約束に含まれるか
- スケジュール: レビュー日やテスト日に影響するか
- コスト: 追加工数はどの程度か
- 品質: 既存タスクの確認密度が下がらないか
- 変更管理: どの項目と入れ替えるか

### 良い返し方

```text
対応自体は可能です。
ただし今 Sprint に入れる場合、既存の改善項目 A を次回へ回す必要があります。
今回優先するか、次回へ送るかの判断をお願いします。
```

## ケース 2: 遅延の予兆が見えた

### 状況

外部チームからの API 仕様確定が遅れており、実装着手が止まりそう。
まだ完全遅延ではないが、金曜レビューに間に合わない可能性が出てきた。

### 弱い反応

- まだ確定していないので言わない
- 自分の中で抱えて様子を見る

### CAPM 的に切る観点

- リスク: 発生可能性と影響度
- スケジュール: どのマイルストーンに影響するか
- ステークホルダー: 誰が先に知るべきか
- コミュニケーション: 何を判断してほしいか

### 良い返し方

```text
API 仕様確定待ちのため、木曜時点で実装着手が遅れるリスクがあります。
このままだと金曜レビューに影響するため、
1. 代替作業へ先に切り替える
2. レビュー日を調整する
の判断を先にお願いします。
```

## ケース 3: 品質認識がずれている

### 状況

実装者は「動いているので完了」と考えているが、顧客側はエラーハンドリングや運用手順まで含めて確認したいと言っている。

### 弱い反応

- 完了定義を後から合わせる
- テストすれば大丈夫とだけ返す

### CAPM 的に切る観点

- 品質: 何をもって完了とするか
- ステークホルダー: 誰の期待値がずれているか
- コミュニケーション: 完了条件を明文化できているか

### 良い返し方

```text
機能動作だけでなく、異常系確認と運用手順整理までを完了条件に含めるか確認したいです。
完了条件をそろえてからレビューに入ります。
```

## ケース 4: 報告はしているのに評価が伸びない

### 状況

毎日進捗は伝えているが、「結局どうしたいのか分からない」と言われる。

### 原因

- 作業報告になっている
- 論点と依頼事項が抜けている

### CAPM 的に切る観点

- コミュニケーション管理
- ステークホルダー期待値
- 意思決定支援

### 改善の型

1. 進捗
2. 論点
3. 放置した場合の影響
4. 判断依頼

## 最後に覚える一文

CAPM の価値は、単語を知ることではなく、案件で起きる揉め事を整理できる観点を増やすことにある。
