# リスク・品質・変更管理

## 1. この章で切れる選択肢

- 問題が起きてから対処する
- リスクを先に見て手を打つ
- 品質をテスト工程だけの話にする
- 変更管理を「断ること」とだけ考える

## 2. 技術の最短定義

リスク管理は、「まだ起きていないが、起きると困ること」を先に洗い、優先して備えることを指す。
品質管理は、「何をもって十分とするか」を先に定義し、それに沿って確認することを指す。
変更管理は、追加要望や優先順位変更を感情ではなく影響整理で扱うことを指す。

この 3 つは、炎上した後の片付け術ではない。
炎上しないように、最初から意思決定を整えるための観点である。

## 3. 今の状態で選ぶ条件

- 技術リスクを感じているが、うまく言語化できない
- 品質基準が曖昧で、レビューや受け入れが人によってぶれる
- 追加要望が入るたびに、現場で吸収する空気になりやすい
- 問題報告が後手になりがち

## 4. 今の状態で選ばない条件

- リスクを悲観論や足踏みと捉える
- 品質をテスト担当だけの責任に押し込む
- 変更管理を対立行為だと思い込む

## 5. ひっかけパターン

- 起きた障害だけをリスクと呼ぶ
- 品質を「バグが少ないこと」だけに狭める
- 変更を全部受けるか全部断るかの二択にしてしまう
- 問題が見えていても、確定してから言おうとして遅れる

## 6. 代表パターン

### 6.1 リスクの見方

| 観点 | 見るべきこと | 実務での言い換え |
|---|---|---|
| 発生可能性 | 起こりやすいか | どれくらい現実的か |
| 影響度 | 起きたらどれだけ困るか | 何が止まるか |
| 対応方針 | 避ける、減らす、受け入れる | 何を先に打つか |

### 6.2 品質の見方

品質は「丁寧に作ること」ではない。
まず必要なのは、受け入れ側と作り手で「何なら通るか」をそろえることである。

| 品質観点 | 最低限そろえること |
|---|---|
| 機能品質 | 期待動作、異常系、境界条件 |
| レビュー品質 | 何をレビュー対象にするか |
| テスト品質 | どこまで確認できれば十分か |
| 運用品質 | 監視、復旧、運用手順の整備 |

### 6.3 変更管理の見方

変更管理は、要望を断るためではなく、変更に伴う影響を透明化するためにある。

変更要求が来たら、最低限次の 5 点で整理する。

1. 変更理由
2. 期待する効果
3. スコープへの影響
4. スケジュール / コスト / 品質への影響
5. 代替案の有無

## 7. 実務へのつなぎ方

### 7.1 リスク報告の言い方

弱い言い方:

```text
ちょっと危ないかもしれません。
```

強い言い方:

```text
外部依存の確定待ちがあるため、木曜時点で着手が遅れるリスクがあります。
回避するなら先に代替案 A を切る必要があります。
```

### 7.2 品質の言い方

- 「品質を上げる」ではなく、「何を満たせば受け入れ可能か」を言う
- 「レビューする」ではなく、「どこを見るレビューか」を言う

### 7.3 変更要求の返し方

- まず受ける姿勢を見せる
- 次に影響範囲を切る
- 最後に、選択肢で返す

例:

```text
対応自体は可能です。
ただし今 Sprint に入れる場合、既存項目 B を後ろへずらす必要があります。
今入れるか、次回へ回すかの 2 案で見てください。
```

## 8. 最後に覚える一文

リスク、品質、変更は、現場努力で隠すものではなく、意思決定可能な形にして早く表に出すための観点である。
