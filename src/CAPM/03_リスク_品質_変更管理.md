# リスク・品質・変更管理

## 1. この章で習得すること

- リスク対応戦略の8種（脅威4種＋機会4種）をPMI定義で覚える
- 定性的リスク分析と定量的リスク分析の違いを切り分ける
- 品質の7つのツールを名前と用途で覚える
- QAとQCの違いを明確にする
- CCBを使った統合変更管理プロセスを理解する

---

## 2. リスク管理

### 2.1 リスクの定義

| 用語 | 定義 |
|---|---|
| リスク | まだ起きていないが、発生するとプロジェクトに影響する不確実な事象 |
| 課題（Issue） | すでに発生している問題。リスクとは区別する |
| 脅威（Threat） | プロジェクトに悪影響を与えるリスク |
| 機会（Opportunity） | プロジェクトに良影響を与えるリスク（PMIはこれも「リスク」として扱う） |
| リスク許容度 | 組織がどの程度のリスクを受け入れられるか |

### 2.2 リスク管理プロセスの流れ

1. リスク管理計画書の作成（どう管理するかを決める）
2. リスクの識別（ブレインストーミング・チェックリスト・SWOT分析等）
3. 定性的リスク分析（優先順位付け）
4. 定量的リスク分析（数値化・必要な場合）
5. リスク対応計画の策定
6. リスクの実施と監視

### 2.3 定性的リスク分析 vs 定量的リスク分析

| 分析 | 目的 | 主なツール |
|---|---|---|
| 定性的リスク分析 | リスクの優先順位付け | 確率・影響マトリクス |
| 定量的リスク分析 | リスクの数値化・金額換算 | モンテカルロシミュレーション、決定木分析 |

**確率・影響マトリクス**：各リスクの「発生確率（高/中/低）」と「影響度（高/中/低）」をマッピングし、対応優先度を決める。

### 2.4 リスク対応戦略（必ず覚える）

#### 脅威（マイナスのリスク）への対応：4種

| 戦略 | 英語 | 意味 | 例 |
|---|---|---|---|
| 回避 | Avoid | リスクが発生しないようにする。計画や条件を変える | 外部APIへの依存をやめ、内製に切り替える |
| 軽減 | Mitigate | 発生確率または影響度を下げる | プロトタイプ検証で技術リスクを早期に洗う |
| 転嫁 | Transfer | リスクを第三者に移す（費用が伴う） | 保険の購入、外部委託、ペナルティ条項 |
| 受容 | Accept | リスクをそのまま受け入れる | コンティンジェンシー予備を積む（能動的受容） |

#### 機会（プラスのリスク）への対応：4種

| 戦略 | 英語 | 意味 | 例 |
|---|---|---|---|
| 活用 | Exploit | 機会が確実に発生するようにする | 得意な技術者を優先的にアサインする |
| 強化 | Enhance | 発生確率または好影響を高める | リソースを追加してスピードを上げる |
| 共有 | Share | 機会を他者と共有して実現確率を上げる | 専門性のあるパートナーと合弁する |
| 受容 | Accept | 積極的に追いかけないが、来たら歓迎する | 特別な対応はしない |

**ポイント**：脅威の「転嫁」は機会の「共有」に対応する。脅威の「軽減」は機会の「強化」に対応する。

### 2.5 コンティンジェンシー予備 vs マネジメント予備

| 予備 | 用途 | 誰が承認するか |
|---|---|---|
| コンティンジェンシー予備 | 識別済みリスクへの対応コスト | PMが承認できる |
| マネジメント予備 | 未識別リスクへの対応コスト | 経営陣/スポンサーが承認する |

## 3. 品質管理

### 3.1 品質の基本概念

| 用語 | 定義 |
|---|---|
| 品質 | 製品・サービスが要件を満たす程度 |
| グレード | 同じ機能目的だが仕様レベルが異なること（低グレードは悪いとは限らない） |
| 精度（Precision） | 繰り返し測定したときの値のばらつきが小さいこと |
| 正確さ（Accuracy） | 真の値にどれだけ近いか |

**重要**：低品質は常に問題だが、低グレードは問題ではない場合がある。

### 3.2 品質保証（QA）vs 品質管理（QC）

| 概念 | 定義 | タイミング | 目的 |
|---|---|---|---|
| 品質保証（QA） | 品質を達成するためのプロセスが適切かを確認する | プロセス全体にわたって継続的に | 予防（プロセスの改善） |
| 品質管理（QC） | 成果物が品質基準を満たしているかを確認する | 成果物の完成後・納品前 | 検査（欠陥の発見と修正） |

**ポイント**：QAはプロセスを見る。QCは成果物を見る。

### 3.3 品質の7つのツール（必ず覚える）

| ツール | 用途 | 覚え方 |
|---|---|---|
| パレート図 | 問題の原因を頻度順に棒グラフで示す。80/20の法則 | 「重要な少数」を特定する |
| 管理図（コントロールチャート） | プロセスが管理限界内にあるかを時系列で監視する | 工程の安定性を確認する |
| ヒストグラム | 測定値の分布を視覚化する | 品質のばらつきを確認する |
| 散布図 | 2つの変数の相関関係を示す | 原因と結果の関係を確認する |
| 特性要因図（フィッシュボーン図） | 問題の原因を体系的に洗い出す | 根本原因分析に使う |
| チェックシート | データを収集・集計するための構造化フォーム | データ収集を標準化する |
| フローチャート | プロセスの流れを図示する | プロセスの全体像を把握する |

### 3.4 管理図の読み方

管理図には「上方管理限界（UCL）」「中心線」「下方管理限界（LCL）」がある。

- データ点が管理限界の外に出た場合 → 原因調査が必要
- 7点連続して中心線の同じ側に並ぶ場合 → 「7点ルール」。異常とみなす
- 仕様限界（顧客要求）と管理限界（プロセス能力）は別物

## 4. 統合変更管理

### 4.1 変更管理の目的

変更管理は「変更を断ること」ではなく、「変更の影響を整理して判断可能な状態にすること」である。

### 4.2 CCB（Change Control Board）

CCBは変更要求を審査・承認・却下する権限を持つ委員会である。

| 役割 | 内容 |
|---|---|
| CCBの権限 | 変更要求の承認・却下・保留の決定 |
| CCBの構成 | PMを含む関係者（スポンサー、顧客代表、チームリード等） |
| PMの役割 | 変更の影響を整理してCCBに提出する。CCBが承認を決定する |

### 4.3 変更管理プロセス（フロー）

```
変更要求の発生
  ↓
変更要求書（Change Request）の作成
  ↓
影響分析（スコープ・スケジュール・コスト・品質への影響）
  ↓
CCBによる審査
  ↓
承認 → ベースライン更新・計画文書更新・実施
却下 → 記録して終了
保留 → 追加情報収集
  ↓
変更ログ・教訓の記録
```

### 4.4 ベースラインの変更

| ベースライン | 変更の承認者 |
|---|---|
| スコープベースライン | CCB |
| スケジュールベースライン | CCB |
| コストベースライン | CCB |

**重要**：PMはベースラインを単独で変更できない。変更には必ずCCBの承認が必要。

## 5. ひっかけパターン

| ひっかけ | 正しい理解 |
|---|---|
| 「転嫁」はリスクをゼロにする | 転嫁はリスクを第三者に移すが、リスク自体はなくならない（保険でも損害は起きる） |
| QAは成果物の品質を検査する | QAはプロセスを確認する。成果物の検査はQC |
| PMが変更を承認する | 変更の承認はCCBが行う。PMは影響分析を提出する側 |
| 管理限界内ならいつも問題なし | 7点連続同側は管理限界内でも異常 |
| 定量的リスク分析は常に必要 | 小規模プロジェクトでは省略されることもある |
| 機会は管理しなくてよい | 機会もリスク管理の対象。活用・強化・共有・受容の戦略がある |

---

## 6. 最後に覚える一文

リスク対応は「脅威への4戦略（回避・軽減・転嫁・受容）」と「機会への4戦略（活用・強化・共有・受容）」を対称で覚える。
品質はQA（プロセス）とQC（成果物）を混同しない。変更はCCBが承認する。
