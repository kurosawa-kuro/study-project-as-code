# anti-cannibalism-rules — フレームワーク衝突を避けるルール（運用）

複数のデファクトフレームワークを併用するとき、役割重複・抽象度衝突（カニバリズム）を起こさないための運用ルール（ADR-004）。

現在のマッピングの正本は [`framework-stack.yaml`](framework-stack.yaml) の `anti_cannibalism` セクション。本書はその読み方と、新しい衝突が見つかったときの解消パターンを定める。

## カニバリズムとは

| 症状 | 例 |
|---|---|
| 役割重複 | RACI と DACI を両方入れて責任者が二重化する |
| 過剰管理 | PMBOK、Scrum、OKR、RAID を全部フル運用して重くなる |
| 抽象度衝突 | MECE、Issue Tree、Pyramid を同じ用途で使って混乱する |
| 技術偏重 | DDD/Clean Architecture ばかりで PM・予算・責任が漏れる |
| 管理偏重 | PMBOK ばかりで技術設計・運用品質が裁けない |

## 解消パターン

新しい衝突を見つけたら、以下のいずれかで解消し、結果を `framework-stack.yaml` の `anti_cannibalism` に追記する。

| パターン | 意味 | 現在の適用例（stack 正本より） |
|---|---|---|
| 主を一つ選ぶ | 同役割なら片方を primary にし他方は使わない | RACI vs DACI |
| 目的で分離する | 見た目が似ていても担当する問いが違うなら両立させる | OKR（目標）vs KPI（計測）、ADR（アーキ判断）vs decision-log（業務/PM 判断） |
| レイヤーで分離する | 原則と成果物、言語と手法など抽象度で棲み分ける | MECE（原則）vs Issue Tree（成果物）、PMBOK（管理言語）vs Scrum（デリバリー手法） |

## ルール

- 衝突の判定と解消案の提示は AI が行ってよいが、マッピングの確定は owner 判断（ADR-004 の役割分担）。
- 解消できない衝突を残したまま stack に両方を high で載せない。
- フレームワーク自体の改変で衝突を回避しない（ADR-003。改変ではなく選択・分離で解く）。
