# PM基礎とコア概念

この章は、CAPMの Domain 1「Project Management Fundamentals and Core Concepts」を固めるための章である。
PMBOK 7th edition の原則・パフォーマンスドメインと、PMBOK 6th edition のプロセスグループ・知識エリアを、試験で迷わない粒度に整理する。

---

## 1. この章で習得すること

- プロジェクト、プログラム、ポートフォリオ、定常業務の違いを説明できる
- PMBOK 7th edition の12原則と8パフォーマンスドメインを把握する
- PMBOK 6th edition の5プロセスグループと10知識エリアを対応づける
- PM、スポンサー、チーム、ステークホルダーの役割差を切り分ける
- 問題、リスク、前提条件、制約条件を混同しない

---

## 2. CAPMで問われる基本概念

| 用語 | 意味 | ひっかけポイント |
|---|---|---|
| プロジェクト | 独自の成果物を生み出す有期的な取り組み | 繰り返し運用ではない |
| 定常業務（Operations） | 継続的に価値を提供する反復業務 | 終了日が明確とは限らない |
| プログラム | 関連する複数プロジェクトをまとめて管理する単位 | 個別プロジェクトより広い便益を見る |
| ポートフォリオ | 戦略目標に沿ってプロジェクトやプログラムを束ねる単位 | 投資配分・優先順位が中心 |
| リスク | まだ起きていない不確実な事象 | 発生前に識別・対応計画を作る |
| 課題（Issue） | すでに起きている問題 | 対応・エスカレーションが必要 |
| 前提条件 | 真実とみなして計画する条件 | 外れるとリスクになる |
| 制約条件 | 選択肢を制限する条件 | 予算・納期・契約条件など |

## 3. PMBOK 7th Edition の構造

### 3.1 12の原則（Principles）

PMBOK 7th editionは「プロセス」より「原則」と「パフォーマンスドメイン」を中心に据えている。
試験では「この原則に基づくと何をすべきか」という形で問われる。

| 原則 | 一言まとめ |
|---|---|
| スチュワードシップ | 誠実・責任感・倫理的に行動する |
| チーム | 協力的なプロジェクトチーム環境を構築する |
| ステークホルダー | ステークホルダーと積極的に関与する |
| 価値 | ビジネス価値と成果に集中する |
| システム思考 | システムとして全体を俯瞰する |
| リーダーシップ | 必要なリーダーシップ行動を示す |
| テーラリング | 状況に合わせてアプローチをカスタマイズする |
| 品質 | 品質をプロセスと成果物に組み込む |
| 複雑性 | 複雑性をナビゲートする |
| リスク | リスク対応を最適化する |
| 適応性・回復力 | 変化に適応し、回復力を構築する |
| 変革 | 意図した将来の状態へ移行できるようにする |

### 3.2 8つのパフォーマンスドメイン

| ドメイン | 焦点 |
|---|---|
| ステークホルダー | ステークホルダーの識別・関与・期待値管理 |
| チーム | チームの構築・文化・パフォーマンス向上 |
| 開発アプローチとライフサイクル | 予測型・アジャイル・ハイブリッドの選択 |
| 計画 | スコープ・スケジュール・コスト・リスクの計画 |
| プロジェクト作業 | 計画の実行・調達・コミュニケーション |
| 納品 | 価値のある成果物の提供と品質確保 |
| 測定 | 進捗の監視・KPI・EVM・是正措置 |
| 不確実性 | リスクと機会の管理・複雑性への対応 |

## 4. PMBOK 6th Edition の参照知識

### 4.1 5つのプロセスグループ

| プロセスグループ | 主な活動 |
|---|---|
| 立上げ（Initiating） | プロジェクト憲章の作成、ステークホルダー特定 |
| 計画（Planning） | 全計画書の作成（スコープ・スケジュール・コスト・リスク等） |
| 実行（Executing） | 計画に従って作業を実施。品質保証・チームマネジメント |
| 監視・コントロール（M&C） | 進捗監視・変更管理・EVMによる分析 |
| 終結（Closing） | プロジェクト正式終了・教訓の記録・契約終結 |

### 4.2 10の知識エリア

| 知識エリア | 主な管理対象 |
|---|---|
| 統合管理 | プロジェクト全体の調整・変更管理・教訓 |
| スコープ管理 | WBS・スコープ定義・スコープコントロール |
| スケジュール管理 | CPM・クリティカルパス・スケジュールベースライン |
| コスト管理 | EVM・予算設定・コストコントロール |
| 品質管理 | QA・QC・品質の7ツール |
| 資源管理 | チーム編成・RACI・リソースアサイン |
| コミュニケーション管理 | コミュニケーション計画・報告 |
| リスク管理 | リスク識別・定性/定量分析・対応計画 |
| 調達管理 | ベンダー選定・契約・調達終結 |
| ステークホルダー管理 | ステークホルダー分析・エンゲージメント |

### 4.3 プロセスグループ × 知識エリア（主要なプロセス）

| 知識エリア | 立上げ | 計画 | 実行 | 監視/コントロール | 終結 |
|---|---|---|---|---|---|
| 統合管理 | プロジェクト憲章作成 | プロジェクト管理計画書作成 | プロジェクト作業の指揮・マネジメント | 統合変更管理 | プロジェクト/フェーズ終結 |
| スコープ管理 | - | スコープ計画・WBS作成 | - | スコープ検証・コントロール | - |
| スケジュール管理 | - | スケジュール計画・CPM開発 | - | スケジュールコントロール | - |
| コスト管理 | - | コスト見積・予算設定 | - | コストコントロール（EVM） | - |
| リスク管理 | - | リスク識別・分析・対応計画 | - | リスクの監視 | - |
| ステークホルダー管理 | ステークホルダー特定 | エンゲージメント計画 | エンゲージメント管理 | エンゲージメント監視 | - |

## 5. 役割と責任

| 役割 | 主な責任 |
|---|---|
| プロジェクトマネジャー（PM） | 制約条件の中でプロジェクトを計画・実行・監視する |
| スポンサー | 資金・権限・主要意思決定を支援する |
| プロジェクトチーム | 成果物を作成し、専門知識を提供する |
| ステークホルダー | プロジェクトに影響を与える、または影響を受ける |

**ポイント**：PMは全てを単独で決める人ではない。影響を整理し、適切なステークホルダーや承認機関へ判断できる形で提示する。

---

## 6. ひっかけパターン

| ひっかけ | 正しい理解 |
|---|---|
| プロジェクトと定常業務を同じものとして扱う | プロジェクトは有期的、定常業務は継続的 |
| リスクを発生後に管理する | 発生後は課題。リスクは発生前に管理する |
| PMBOK 7thだけ覚えればよい | CAPMではプロセスグループや知識エリアも参照知識として必要 |
| PMが全て承認する | 変更承認はCCBやスポンサーなど権限者が行う |

---

## 7. 最後に覚える一文

CAPMの基礎問題は、用語暗記ではなく「どの概念がどの役割を持つか」を問う。
プロジェクト・定常業務・リスク・課題・役割責任を混同しないことが、シナリオ問題の土台である。
