# アジャイルとハイブリッド

この章は、CAPMの Domain 3「Agile Frameworks/Methodologies」を固めるための章である。
ScrumやKanbanの用語だけでなく、どの状況で予測型・アジャイル・ハイブリッドを選ぶかを判断できる状態を目指す。

---

## 1. この章で習得すること

- アジャイルと予測型の違いを説明できる
- Scrumの役割・アーティファクト・イベントを混同しない
- KanbanのWIP制限、リードタイム、サイクルタイムを説明できる
- ハイブリッドを選ぶ状況を判断できる
- アジャイルの計画・追跡・タスク管理を試験シナリオで選べる

---

## 2. アジャイルと予測型の比較

| 項目 | 予測型（Predictive） | アジャイル |
|---|---|---|
| 計画 | 最初に全体計画を固める | 短いサイクルで繰り返し計画する |
| 変更への対応 | 変更管理プロセスで制御 | 変化を前提に優先順位を見直す |
| 成果物の納品 | プロジェクト終了時または大きな節目で納品 | 短いサイクルで継続的に納品 |
| 向いている状況 | 要件が安定・明確 | 要件が変わりやすい・不確実 |
| 進捗の見方 | ベースラインとの差異を見る | 完成した価値・フロー・ベロシティを見る |

## 3. Scrumの構成要素

### 3.1 Scrumの役割

| 役割 | 責任 |
|---|---|
| プロダクトオーナー（PO） | プロダクトバックログの優先順位を管理する。What（何を作るか）を決める |
| スクラムマスター（SM） | Scrumプロセスのファシリテーター。障害を取り除く |
| 開発チーム | スプリント内の作業を実施する自己組織化チーム |

### 3.2 Scrumのアーティファクト

| アーティファクト | 内容 |
|---|---|
| プロダクトバックログ | 製品に必要な全機能・要件のリスト。POが管理する |
| スプリントバックログ | そのスプリントで実施するタスクのリスト |
| インクリメント | スプリントで完成した動作可能な成果物の累積 |

### 3.3 Scrumのイベント

| イベント | 目的 | タイミング |
|---|---|---|
| スプリント計画 | スプリントで何を作るかを決める | スプリント開始時 |
| デイリースクラム | チームの状況共有・障害特定 | 毎日 |
| スプリントレビュー | 完成したインクリメントをデモしフィードバックを得る | スプリント終了時 |
| スプリントレトロスペクティブ | プロセスを振り返り改善点を特定する | スプリントレビュー後 |
| バックログリファインメント | プロダクトバックログを整理・詳細化・見積もる | スプリント中 |

### 3.4 Scrumのキーワード

| 用語 | 意味 |
|---|---|
| スプリント | 1〜4週間の反復サイクル |
| ベロシティ | 1スプリントで完成できる作業量の実績値 |
| バーンダウンチャート | 残作業量を時系列で示すグラフ |
| 定義完了（DoD） | 「完了」の共通基準 |
| ユーザーストーリー | 「誰として、何をしたい、なぜなら」の形で表す要求 |

## 4. Kanbanの基礎

| 概念 | 内容 |
|---|---|
| Kanbanボード | タスクを「To Do / In Progress / Done」等の列で管理する可視化ツール |
| WIP制限 | 同時進行中の作業数に上限を設ける。フロー改善のための仕組み |
| フロー | タスクが左（未着手）から右（完了）へスムーズに流れること |
| リードタイム | タスクが要求されてから完成するまでの時間 |
| サイクルタイム | タスクの作業開始から完成までの時間 |

**ポイント**：KanbanはScrumのような固定スプリントを必須としない。WIP制限で作業の詰まりを見つけ、流れを改善する。

---

## 5. ハイブリッドアプローチ

ハイブリッドは、予測型とアジャイルを組み合わせたアプローチである。

| 状況 | 推奨アプローチ |
|---|---|
| 要件が明確・変化が少ない | Predictive（予測型） |
| 要件が不明確・変化が多い | Agile |
| 一部固定・一部変化あり | Hybrid |

**例**：規制対応や契約範囲は予測型で管理し、UIや利用者体験はアジャイルで反復的に改善する。

## 6. 演習問題

**問題A**

Scrumでプロダクトバックログの優先順位を決める責任を持つのは誰か。

- A. スクラムマスター
- B. 開発チーム
- C. プロダクトオーナー
- D. ステークホルダー委員会

> **正解：C**
> プロダクトバックログの優先順位管理はプロダクトオーナーの責任。スクラムマスターはプロセスファシリテーター。

---

**問題B**

Kanbanで「WIP制限」を設ける主な目的はどれか。

- A. チームメンバーの業務量を均等にする
- B. 同時進行タスクを制限してフローを改善し、リードタイムを短縮する
- C. スプリント内の作業計画を安定させる
- D. バックログの優先順位を決める

> **正解：B**
> WIP制限は作業の流れを改善するための仕組み。CはScrumの説明。

---

**問題C**

スプリントレトロスペクティブの目的はどれか。

- A. 完成したプロダクトをデモしてフィードバックを得る
- B. 次のスプリントの作業を計画する
- C. チームのプロセスを振り返り、改善点を特定する
- D. プロダクトバックログを整理する

> **正解：C**
> レトロスペクティブはプロセス改善が目的。成果物のデモはスプリントレビュー。

## 7. ひっかけパターン

| ひっかけ | 正しい理解 |
|---|---|
| アジャイルでは計画しない | アジャイルも計画する。ただし短期サイクルで繰り返す |
| スクラムマスターがバックログを管理する | バックログ管理はプロダクトオーナー |
| ベロシティは常に一定 | ベロシティはスプリントごとに変動する。実績の平均で予測する |
| KanbanにはScrumと同じスプリントがある | Kanbanは固定スプリントを必須としない |
| 予測型は変更を受け付けない | 変更管理プロセスを通じて変更を管理する |

---

## 8. 最後に覚える一文

アジャイル問題は用語暗記だけでは足りない。
Scrumの責任分界、Kanbanのフロー改善、そして予測型・アジャイル・ハイブリッドの選択条件をシナリオで判断できることが合格の条件である。
